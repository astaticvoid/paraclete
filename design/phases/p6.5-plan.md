## Commit 1 — EnvelopeNode Looping AD fix

**File:** `paraclete-nodes/src/envelope.rs`

### Problem

The P6 report states: *"Looping AD: resets to Idle (→ next gate re-triggers)
after Decay completes."*

The spec says Looping AD transitions to Attack autonomously, without a gate.
It loops. Going to Idle means the envelope stops after one cycle and waits
for the next gate rising edge — that is AD mode with a different name, not
Looping AD.

### Root cause

In the Decay match arm, the Looping AD branch likely reads:

```rust
_ => { // Looping AD
    if self.value < 1.0e-5 {
        self.value = 0.0;
        self.phase = EnvPhase::Idle;   // ← wrong
    }
}
```

### Fix

```rust
_ => { // Looping AD — restart Attack without gate
    if self.value < 1.0e-5 {
        self.value = 0.0;
        self.phase = EnvPhase::Attack; // ← correct
    }
}
```

The envelope autonomously loops Attack → Decay → Attack. The gate input
is still wired and still works: a gate rising edge always retriggers from
Attack regardless of mode. Looping AD just doesn't need the gate to keep
cycling.

### Test to add

```
envelope_looping_ad_no_gate_cycles_continuously
    Drive EnvelopeNode with block_size × N samples, gate held at 0.0
    (unconnected / all-zeros) throughout.
    After the first Decay completes (one cycle duration = attack_s + decay_s),
    output must remain non-zero for at least one further full cycle.
    Verifies autonomous looping without any gate signal.
```

This test is distinct from the existing `envelope_looping_ad_restarts_after_decay`,
which should be inspected: if it passes a gate-high signal, it is testing the wrong
behavior. Retain it (gate-high retrigger in looping mode is also valid) but add
the no-gate test as the primary looping correctness assertion.

### Done criterion

`envelope_looping_ad_no_gate_cycles_continuously` passes. All existing envelope
tests continue to pass.

## Commit 2 — LadderFilterNode parameter names

**File:** `paraclete-nodes/src/ladder.rs`

### Problem

`LadderFilterNode` declares its parameters as `"ladder_cutoff"`,
`"ladder_resonance"`, and `"ladder_drive"`. These prefixed names are
inconsistent with the established convention (see ADR-019 amendment):

- `FilterNode` uses `"cutoff"` and `"resonance"`
- `DistortionNode` uses `"drive"`
- `AnalogEngine` and `FmEngine` use `"drive"`, `"decay"`, `"tune"`

A hardware encoder mapped to `id_for_name("cutoff")` reaches `FilterNode`
but not `LadderFilterNode`. This silently defeats the universal parameter
control model. The prefixed names will be published to crates.io as-is
if not fixed now; after that point, changing them is a semver-breaking
change with a migration story.

### Fix

```rust
// Before:
pub const PARAM_LADDER_CUTOFF:    u32 = id_for_name("ladder_cutoff");
pub const PARAM_LADDER_RESONANCE: u32 = id_for_name("ladder_resonance");
pub const PARAM_LADDER_DRIVE:     u32 = id_for_name("ladder_drive");

// After:
pub const PARAM_LADDER_CUTOFF:    u32 = id_for_name("cutoff");
pub const PARAM_LADDER_RESONANCE: u32 = id_for_name("resonance");
pub const PARAM_LADDER_DRIVE:     u32 = id_for_name("drive");
```

The public constant names (`PARAM_LADDER_CUTOFF` etc.) are unchanged —
only the hash inputs change. This lets call sites distinguish `FilterNode`
and `LadderFilterNode` constants at the type level while the on-wire IDs
unify for hardware addressing.

Update `capability_document()` param name strings to match:
```rust
ParamDescriptor { name: "cutoff",    ... }
ParamDescriptor { name: "resonance", ... }
ParamDescriptor { name: "drive",     ... }
```

### Tests

No new tests. Update any test that asserts specific param name strings.
All existing functional tests (`ladder_attenuates_above_cutoff` etc.)
pass unchanged — the hash change only affects cross-node addressing.

### Done criterion

`id_for_name("cutoff")` on `LadderFilterNode` matches `id_for_name("cutoff")`
on `FilterNode`. A `CMD_BUMP_PARAM` command with `id_for_name("cutoff")` and
a delta value reaches both nodes when broadcast. Confirm with an integration
test or a quick manual check against the constant values.

## Commit 3 — Signal port executor allocation

**Files:** `paraclete-runtime/src/executor.rs`,
`paraclete-runtime/src/configurator.rs`

### Problem

`EnvelopeNode`, `LfoNode`, and `OscillatorNode` declare signal output ports
(Modulation, Logic) and call `output.mod_output_mut()` / `output.logic_output_mut()`
in `process()`. The executor currently only pre-allocates buffers for
Audio/Mono output ports. Non-audio signal ports have no backing buffers,
so these nodes cannot be registered in the graph — their tests construct
`SignalOutputSlot` manually and are not integration tests.

The P6 report acknowledged this: *"signal port pre-allocation (SignalOutputSlot)
deferred to P7."* Moving it here rather than mid-P7 prevents the CLAP inner-graph
work from building on this gap.

### Design

Following the pattern of P4.5 Fix 4 (audio routing), add signal routing to
the executor.

#### `NodeExecutor` additions

```rust
// paraclete-runtime/src/executor.rs

pub struct NodeExecutor {
    // existing fields unchanged

    // Pre-allocated buffers for non-audio signal output ports.
    // signal_output_bufs[slot_idx][port_id] = Vec<f32> of block_size samples.
    // Zeroed at the start of each process cycle.
    signal_output_bufs: Vec<HashMap<u32, Vec<f32>>>,

    // Pre-computed routing: for a given slot's signal input port,
    // the (upstream_slot_idx, upstream_port_id) providing its data.
    // If absent, the port is unconnected (reads zeros).
    // signal_input_routes[slot_idx][port_id] = (src_slot, src_port_id)
    signal_input_routes: Vec<HashMap<u32, (usize, u32)>>,
}
```

#### `configurator.rs` changes

In `build_executor()`, after the existing audio routing pass, add a
signal routing pass:

```rust
// For each node slot, find all signal output ports (Modulation, Logic,
// Cv, Pitch, Phase) and allocate a Vec<f32> of block_size samples.
for slot_idx in 0..slots.len() {
    let mut port_bufs: HashMap<u32, Vec<f32>> = HashMap::new();
    for port in nodes[slot_idx].ports() {
        if port.direction == PortDirection::Output
           && is_signal_port_type(port.port_type)
        {
            port_bufs.insert(port.id, vec![0.0f32; block_size]);
        }
    }
    signal_output_bufs.push(port_bufs);
}

// For each edge where src_port_type is a signal type, record the route
// on the destination slot's signal_input_routes.
for edge in graph.edge_references() {
    let meta = edge.weight();
    if is_signal_port_type(meta.port_type) {
        let dst_slot = topology[edge.target()];
        let src_slot = topology[edge.source()];
        signal_input_routes[dst_slot]
            .insert(meta.dst_port, (src_slot, meta.src_port));
    }
}
```

```rust
fn is_signal_port_type(pt: PortType) -> bool {
    matches!(pt, PortType::Modulation | PortType::Logic
                 | PortType::Cv | PortType::Pitch | PortType::Phase)
}
```

#### `executor.rs` process loop changes

```rust
// At the start of each cycle, zero all signal output buffers
for slot_bufs in &mut self.signal_output_bufs {
    for buf in slot_bufs.values_mut() {
        buf.fill(0.0);
    }
}

// Before each slot's process() call, build ProcessOutput with access
// to that slot's signal output buffers (alongside existing audio bufs).
// After process() returns, the buffers contain the node's output.
```

`ProcessOutput::mod_output_mut(port_id)` and `logic_output_mut(port_id)` look
up `signal_output_bufs[current_slot][port_id]` and return `&mut [f32]`.

`ProcessInput::modulation(port_id)` and `logic(port_id)` check
`signal_input_routes[current_slot][port_id]` for a route; if found,
return `&signal_output_bufs[src_slot][src_port_id]`; if absent, return
`&SILENCE[..block_size]`.

Topological order guarantees the upstream slot has already run in the
same cycle, so its signal output buffers are valid when downstream reads them.
This is the same guarantee that makes audio routing correct.


### Notes

- `unsafe impl Send` on `NodeExecutor` may need updating — verify that
  `HashMap<u32, Vec<f32>>` is `Send` (it is, since Vec<f32> is Send and
  HashMap keys/values are both Send). The existing raw pointer scratch
  buffer concern from P4.5 Fix 4 does not extend to this change.
- The `send` constraint on `NodeExecutor` was previously justified by the
  audio thread transfer. The new signal buffers are owned data, no raw
  pointers, so no additional `unsafe` is needed.
