> **Implementation blueprint.** These are the contracts P9.5 implementation
> must satisfy. Do not deviate without updating the roadmap/ADRs first.
>
> **P9.5 deliverable:** Device emulation & test harness — every feature of the
> supported surfaces is exercisable from the terminal, no physical hardware
> required to develop or test. Unblocks P10/P11 surface verification.
> **Last updated:** June 2026
> **Baseline:** P9 shipped — 391 tests (389 distinct), 0 failures
> **Depends on:** p0–p9; the existing `LaunchpadEmulator` (`paraclete-hal/src/emulator/`).
> **References:** roadmap P9.5 scope; `controller-strategy` memory; ADR-019
> (universal parameter control); ADR-026 (TUI / context); p10-interfaces (Commit 5

## Why this phase

Audit (June 2026): only the Launchpad has a software emulator, and it reaches
just **24 of 64 pads** (rows 0–2 via `Q/A/Z` letter rows), with **no scene
buttons, no control row, fixed velocity, no RGB feedback** in the keyboard
mapping. P10's pages/pattern-switching (scene buttons, all 8 tracks) and P11's
mutes/fills are bound to controls the emulator can't reach — so without this work
they are hardware-only to verify. P9.5 makes the full surface keyboard-driven and
adds the encoder/note/headless paths the rest of the roadmap needs.

**Hard constraint (controller-strategy memory):** parameter control is built on
**relative encoders** (`CMD_BUMP_PARAM`). Absolute pots are disqualified for this
instrument. The emulator's virtual encoders therefore emit **relative deltas**
(`HardwareEvent::EncoderChanged { delta }`), never absolute positions.

## Control ID map (emulator surface)

| Range | Control | Event |
|-------|---------|-------|
| 0–63 | 8×8 grid pads (row-major, `row*8 + col`) | `PadPressed`/`PadReleased` |
| 64–71 | Scene buttons (right column, page select) | `PadPressed`/`PadReleased` |
| 72–79 | Top control row (modes / navigation) | `PadPressed`/`PadReleased` |
| 90–97 | Virtual encoders (Commit 3, `EncoderBehaviour::Relative`) | `EncoderChanged { delta }` |

Scene/control are modelled as pads-by-id (the real device uses CC buttons; the
gateway/profile abstracts by id range, as today). Profiles branch on id range.

# Commit 1: Full Launchpad emulator (priority / p10-C5 gate)

**Crate:** `paraclete-hal` (`src/emulator/`)

**Problem:** `key_to_pad()` maps only 24 keys (pads 0–23). Scene buttons render
but have no keys; there is no control row; rows 3–7 are unreachable.

**Fix:** a **track-cursor** keyboard scheme — the number row selects which track
row the step keys edit, so all 64 pads are reachable with a stable mental model
("select a track, edit its 8 steps"), plus dedicated rows for scene and control.

## 1.1 Keyboard scheme

| Key(s) | Action |
|--------|--------|
| `1`–`8` | Select active track row (0–7). Cursor only — emits no event. |
| `Q W E R T Y U I` | Toggle the 8 step pads (col 0–7) **in the active row** → pad id `active_row*8 + col` |
| `A S D F G H J K` | Scene buttons 0–7 → ids 64–71 |
| `Z X C V B N M ,` | Control-row buttons 0–7 → ids 72–79 |
| `Esc` / `Ctrl-C` | Quit (unchanged) |

This **replaces** the legacy `Q/A/Z = rows 0/1/2` mapping. Update the keyboard-
controls sections of `README.md` and `CLAUDE.md` accordingly (do this in the
same commit — docs match behaviour).

## 1.2 Pure mapping (unit-testable)

Refactor `terminal.rs` so the key→target mapping is a pure function, separate
from crossterm I/O and emulator state:

```rust
pub(super) enum KeyTarget {
    RowSelect(u8),  // 0-7
    Step(u8),       // col 0-7
    Scene(u8),      // 0-7  -> id 64+n
    Control(u8),    // 0-7  -> id 72+n
}
pub(super) fn key_to_target(code: KeyCode) -> Option<KeyTarget>;
```

The emulator (`mod.rs`) applies targets against `active_row`:
- `RowSelect(r)` → `self.active_row = r` (re-render; no event).
- `Step(c)` → id `= active_row*8 + c`; on press emit `PadPressed`, on release `PadReleased`.
- `Scene(n)`/`Control(n)` → id `64+n` / `72+n`; press/release events.

**Press/release id tracking:** store `held: HashMap<KeyCode, u32>` (key → the id
emitted on its press) so a release emits the correct id even if `active_row`
changed meanwhile. Render's pressed set = `held.values()`.

## 1.3 Surface + render

- `build_launchpad_surface()` gains the **8 control-row pads (ids 72–79)** in
  addition to the existing 64 grid + 8 scene.
- `render()` shows the control row, the 8×8 grid with the **active row marked**
  (e.g. `>` gutter), pressed pads as `[#]`, and the scene column. Header shows
  the active row and a one-line key legend.

## 1.4 Mode scaffold

Add `enum EmuMode { Grid, Encoder, Piano }` (default `Grid`) and a toggle key
(`Tab` cycles modes; mode shown in the header). Commit 1 implements **Grid** only;
Encoder/Piano are wired in Commits 3/4. In non-Grid modes Commit 1 ignores
control keys (no-op) — no behavioural surprise.

## 1.5 Tests (`paraclete-hal`)

- `key_to_target_maps_number_row_to_row_select`
- `key_to_target_maps_qwerty_to_steps` (Q→0 … I→7)
- `key_to_target_maps_home_row_to_scene` and `bottom_row_to_control`
- `step_id_uses_active_row` — with `active_row = 3`, `Step(2)` → id 26
- `release_emits_pressed_id_after_row_change` — press a step on row 1, change to
  row 5, release → `PadReleased` carries the row-1 id
- `unmapped_key_returns_none`
- `surface_has_64_grid_8_scene_8_control` (80 pad controls total)

# Commit 2: RGB LED feedback

**Crates:** `paraclete-hal`; `paraclete-runtime` (routing only, if not already wired)

**Goal:** the emulator shows LED colours driven by the app/profiles — so you can
see Elektron-style colour coding, the current-step sweep, and (P10) cued-pattern
blink without hardware.

- Add `led: HashMap<u32, RgbColor>` to the emulator. Implement
  `update_output(&HardwareOutput)` to apply each `LedUpdate { control_id, color }`.
  (`update_output` is the legacy audio-thread path; the emulator already uses it
  and both it and `render()` run in the executor, so no locking needed.)
- `render()` colourises each control from `led` (nearest ANSI/24-bit colour via
  crossterm `style`), falling back to off/dim when unset. Pressed state overlays.
- **Verify the app routes LED output to the emulator (device id 101).** If the
  legacy `HardwareOutput` is not delivered to the emulator today, wire it: the
  profile LED writes (`deliver_script_output`) must reach the registered fallback
  device. This is the only `paraclete-runtime` touch; keep it minimal.

**Tests:** `update_output_sets_led_color`; `render_colorizes_from_led_map`
(assert the emulator stores the colour; visual render is play-tested).

# Commit 3: Virtual relative encoders + `CMD_BUMP_PARAM` acceleration

**Crates:** `paraclete-hal`, `paraclete-node-api`, `profiles/`

**Goal:** test contextual parameter control from the keyboard, exercising the
**relative** path that real controllers (Intech EN16 / MFT / OXI) will use.

## 3.1 Emulator Encoder mode

- Surface gains **8 virtual encoders, ids 90–97**, `EncoderBehaviour::Relative`,
  `has_push: false`.
- In `EmuMode::Encoder`: `1`–`8` select the active encoder; `,`/`.` (or `←`/`→`)
  emit `EncoderChanged { id: 90+sel, value: 0, delta: ∓1 }`; hold a modifier for
  a larger delta (coarse). No absolute position is ever sent.

## 3.2 Acceleration / fine-step (profile or param layer)

- The profile maps `EncoderChanged.delta` → `CMD_BUMP_PARAM` (type_id 1). Apply
  **acceleration**: scale the applied step by recent turn speed (fast → coarse,
  slow → fine 0.0-level tweaks). With relative encoders the contextual remap is
  seamless — **no pickup/soft-takeover** (that path is for absolute pots and is
  not implemented).
- Document the canonical relative→bump mapping in `profiles/` so EN16/MFT/Digitakt
  share it (per-device CC numbers differ; the delta→bump logic is common).

## 3.3 Doc-comment fix

`EncoderBehaviour::Relative` in `node-api/src/hardware.rs` says *"Used by Digitakt
and XL Mk3."* The XL Mk3 is absolute-only in custom modes — **remove the "XL Mk3"
reference** (it's factually wrong; see controller-strategy memory).

**Tests:** `encoder_mode_emits_relative_delta` (never absolute);
`encoder_select_changes_target_id`; `bump_accel_scales_step_with_speed`.

# Commit 4: Piano mode + headless input injection

**Crates:** `paraclete-hal`, `paraclete-app`

## 4.1 Piano mode (Keystep role)

- `EmuMode::Piano`: a chromatic keyboard layout (e.g. `Z S X D C V G B H N J M`
  = C…B, `,` = next C; `A`/`;` octave down/up) emits **`Event::Midi2` note-on/off**
  (not `HardwareEvent` — notes go to the sequencer/synth path, like the Keystep).
- Fixed velocity (reuse `KEY_VELOCITY`); release → note-off.

## 4.2 Scripted / headless input injection (CI)

- Define a tiny line format for a device-event script (no TTY required):
  `cycle event` per line, e.g. `0 pad 5 on`, `2 pad 5 off`, `4 enc 90 +3`,
  `8 note 60 on`. A loader parses it into `(cycle, Event)` pairs.
- Drive it through the **existing executor injection API** (`inject_event_for_node`,
  `inject_node_command`) — no new runtime surface. Provide an integration-test
  helper in `paraclete-app/tests/` that runs N cycles, injects the scripted
  events at their cycles, and asserts resulting state-bus values.
- This is the mechanism P10/P11 integration tests use to cover scene-button page
  select, cued switching, etc. (p10-interfaces §5.4) without hand-testing.

**Tests:** `piano_mode_emits_midi_note_on_off`; `input_script_parses_lines`;
`injected_script_drives_executor_state` (one end-to-end: inject pad toggles, assert
`/state/steps`).

## Definition of Done

- All commits merged in order; code review before each (`feedback-p8-workflow`).
- `cargo test --workspace` green; `cargo clippy` clean; `paraclete-nodes` /
  `paraclete-node-api` portability checks pass.
- `README.md` + `CLAUDE.md` keyboard-control sections updated to the track-cursor
  scheme (Commit 1).
- **Play-test (interactive, by the user):** in a real terminal, reach all 8
  tracks + scene + control from the keyboard; see LED colours; nudge a param via
  Encoder mode; play notes via Piano mode. (Crossterm raw-mode can't be driven in
  CI — the headless *injection* path (Commit 4) covers automated coverage; the
  *interactive* feel is user-verified.)
- `design/phases/p9.5-report.md` written; roadmap P9.5 marked Complete.

## Out of scope

- Faders (the emulator stays encoder-focused; faders are absolute by nature and
  the vision leans on encoders — add later if needed).
- A standalone XL/MFT/EN16 device node — those are real `midir` devices configured
  per the controller-strategy memory, not emulated. The virtual encoders here are
  for *keyboard* testing of the relative path, not a model of a specific box.
- Multi-device simultaneous emulation (one emulated surface at a time).
