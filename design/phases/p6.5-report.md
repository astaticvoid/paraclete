Commit 1 — EnvelopeNode Looping AD fix (paraclete-nodes/src/envelope.rs)

  Problem: Looping AD mode (env_mode=2) required a gate pulse to re-enter Attack
  after each Decay completed. Without a gate, the envelope sat in Idle and produced
  silence. This made Looping AD behave identically to AD mode, defeating its purpose.

  Root cause: The Idle match arm was a no-op (`EnvPhase::Idle => {}`). For Looping
  AD, the spec requires autonomous cycling — the envelope restarts without a gate
  after the Decay phase completes. The Decay arm correctly transitioned to Attack,
  but if the envelope ever returned to Idle (e.g. at activation, or any future path
  that sets phase=Idle), mode=2 would stop.

  Fix: Added autonomous start in the Idle arm:
    ```rust
    EnvPhase::Idle => {
        if mode == 2 { self.phase = EnvPhase::Attack; }
    }
    ```
  This means Looping AD begins its first cycle on the first process() call, with no
  gate needed. A gate rising edge still retriggers (gate_rose path sets phase=Attack
  regardless of mode — no change there).

  Deviation from plan: The plan described the root cause as "Decay arm transitions
  to Idle instead of Attack." The actual Decay arm already transitioned to Attack
  correctly. The real gap was that the Idle arm didn't handle autonomous restart, so
  any future Idle entry (including on activation) would stall the loop. Fix addresses
  the correct root cause.

  Test added:
    envelope_looping_ad_no_gate_cycles_continuously
      8 blocks, gate=0 throughout. Counts peaks > 0.9 in output. Asserts >= 2 peaks,
      confirming the envelope cycles autonomously.

  Existing test (envelope_looping_ad_restarts_after_decay) retained unchanged. It
  uses a gate pulse to start the first cycle; after the fix, the envelope would loop
  without it, but the gate retrigger is also valid behavior.


Commit 2 — LadderFilterNode parameter rename (paraclete-nodes/src/ladder.rs)

  Problem: LadderFilterNode declared parameters as "ladder_cutoff", "ladder_resonance",
  "ladder_drive". The `id_for_name()` hash model only provides cross-node generic
  control if parameters representing the same concept share the same name string.
  A hardware encoder or profile script mapped to id_for_name("cutoff") would reach
  FilterNode but not LadderFilterNode. ADR-019 (amended P6.5) formalises this as the
  canonical naming convention.

  Fix: All three parameter name strings replaced:
    "ladder_cutoff"    → "cutoff"
    "ladder_resonance" → "resonance"
    "ladder_drive"     → "drive"

  Both the `name:` field in ParamDescriptor and the `lad("...")` hash call sites
  updated. The public constant names (PORT_AUDIO_IN, PORT_CUTOFF_MOD, etc.) and the
  `lad()` helper function name are unchanged — only the hash inputs changed.

  Result: id_for_name("cutoff") now returns the same ID for both FilterNode and
  LadderFilterNode. A CMD_BUMP_PARAM with that ID reaches both nodes generically.

  No new tests. All 5 existing functional tests (ladder_attenuates_above_cutoff, etc.)
  pass unchanged — the hash change only affects cross-node addressing, not DSP.


Commit 3 — Signal port executor allocation
  (paraclete-runtime/src/executor.rs, paraclete-runtime/src/configurator.rs)

  Problem: The executor passed `signal_inputs: &[]` and `signal_outputs: &mut []` to
  every node's process() call. Nodes that call output.mod_output_mut() or
  output.logic_output_mut() (EnvelopeNode, LfoNode, OscillatorNode) panicked when
  registered in the executor because find_output() panics on a missing port slot.
  These nodes were therefore not wireable in graphs; their tests constructed
  SignalOutputSlot manually.

  Design: Follows the audio routing pattern established in P4.5 Fix 4.

  configurator.rs additions:
    is_signal_port_type(pt: PortType) -> bool
      Returns true for Modulation, Logic, Cv, Pitch, Phase.
    port_type_to_signal_kind(pt: PortType) -> SignalPortKind
      Converts PortType to the matching SignalPortKind variant.

    In build_executor(), first pass (edge routing):
      New match arm for signal port types. For each edge where src_port_type is a
      signal type, records (dst_port, kind, src_slot, src_port) into
      signal_input_routes[dst_pos].

    In build_executor(), second pass (after ordered is built):
      For each node's signal Output ports, allocates a Vec<f32> of block_size
      samples into signal_output_bufs[slot_pos].

  executor.rs additions:
    New struct fields:
      signal_output_bufs: Vec<Vec<(u32, SignalPortKind, Vec<f32>)>>
        Pre-allocated output buffers per slot, keyed by (port_id, kind, buffer).
        Zeroed at the start of each slot's process() call.
      signal_input_routes: Vec<Vec<(u32, SignalPortKind, usize, u32)>>
        Pre-computed routing: (dst_port, kind, src_slot, src_port).
        Built at build_executor() time; never mutated at runtime.
      signal_out_scratch: Vec<SignalOutputSlot>
        Per-cycle scratch; cleared and rebuilt each slot. No audio-thread allocation.
      signal_in_scratch: Vec<SignalInputSlot>
        Same for inputs.

    In process() per-slot loop:
      Before the slot mutable borrow, two scoped blocks use direct field borrows
      (Rust permits disjoint field borrows simultaneously) to build signal_out_scratch
      and signal_in_scratch. Raw pointer slices are extracted after the scopes release,
      following the established pattern for audio_input_scratch.
      ProcessInput.signal_inputs and ProcessOutput.signal_outputs now pass the
      pre-built slices via unsafe from_raw_parts / from_raw_parts_mut.

  Topological order guarantee: signal_input_routes[dst] always references slots with
  smaller indices (processed before dst). Same guarantee as audio routing.

  Tests added (runtime_integration.rs, 3):
    signal_port_executor_allocates_mod_output
      Registers a ModOutputNode (which calls output.mod_output_mut(0)). Runs
      exec.process(). Asserts no panic — confirming the port slot was allocated.

    signal_routing_mod_output_reaches_mod_input  ← done criterion
      ModOutputNode (value=0.5) connected to ModInputNode. After exec.process(),
      ModInputNode's received value is 0.5 ± 1e-6.

    signal_port_unconnected_returns_silence
      ModInputNode registered alone (no upstream). input.modulation(0) returns
      SILENCE[..block_size]; first sample is 0.0.

  Deviation from spec: spec used HashMap<u32, Vec<f32>> for signal_output_bufs and
  HashMap<u32, (usize, u32)> for signal_input_routes. Implemented as Vec of tuples
  instead. At typical port counts (1–2 signal outputs per node), linear scan over a
  Vec is faster than a HashMap lookup and avoids HashMap construction overhead. The
  logical contract is identical.
