## Findings (live)

- ~~F1: session token not randomized across restarts~~ — RETRACTED. The token
  is deliberately persisted to `.antiphon-token` and reused so reconnecting
  clients aren't bounced (`main.rs:401`). Working as designed, not a finding.
- E1 (ergonomics, resolved same-session): the 32-hex token made the URL
  un-tappable. Added `--open` flag (trusted-LAN, opt-in): drops the token so the
  URL is a bare `http://nimbus.local:7274/`. Default still tokened. The web
  client already sends `token:""` when `?t=` is absent, so no client change.
  Verified: empty-token handshake accepts; mDNS host serves. Tradeoff (any LAN
  device can control) is explicit and accepted for the home network.

- **F2 (headline): the Theoria client is an unlabeled test harness, not a
  legible instrument.** Driver spent the first stretch unknowingly in trigger
  mode, eventually reverse-engineered how to sequence one sound. Root cause
  (confirmed in code): `Grid.tsx` sends raw pad ids; there is **no client-side
  mode concept** — trigger-vs-sequence is server-side Rhai state with no visible
  indicator. Only labeled affordance is the transport-bar track buttons. Nothing
  on the surface communicates what a pad does or what mode is active.
  - Impact: W1's plumbing is proven, but the surface is not evaluable *as an
    instrument* by a human without a guide.
  - Direction: front-load UX legibility before more protocol surface. Minimum
    bar: (a) visible current-mode indicator, (b) grid regions/pads labeled by
    function, (c) selected-track ↔ grid relationship shown, (d) param names on
    the encoder row (already partly there).
- F3 (minor): `[deliver] no output handle for device 101` log spam — phantom
  Launchpad-emulator LED updates dropped when no 101 sink exists. Suppress/gate
  when no emulator surface is registered.

- F4: **state mirror drops all Text values** (`lib.rs:134`, `Text => continue`).
  The step bitfield (`/state/steps`) and `track_name` never reach the tablet on
  the state channel — the grid reflects programmed steps via the **LED channel**
  only. Fine for an LED grid; but a client cannot obtain the *semantic* pattern
  (which steps, what notes) to build a richer sequencer view. If W2 wants that,
  the mirror needs a Text-capable state frame (or a dedicated pattern message).
- F5: transport bar renders **8 track buttons but the default (no-yaml) graph
  has only 4 sequencers** (10–13). Buttons 5–8 target non-existent nodes — dead.
  Either drive the count from the welcome topology or ship an 8-track default.
- F6 (positive): programming path verified end-to-end — `node_cmd` →
  `CMD_TOGGLE_STEP` routes to the sequencer and the LED channel reflects it.
  The grid shows **one track's 16 steps at a time** (pads 0–15); switching the
  displayed track is via the transport track buttons, not a grid gesture.
- Criterion 6 (RTT): tablet median **~14–15 ms** over wifi; server pong floor
  **0.10 ms** (loopback ×30). Latency is entirely wifi; well within playable.

- F7 (**CORRECTED — impact overstated**): `launchpad.rhai` hardcodes every
  `set_led` to `LP_DEVICE_ID`, never `THEORIA_DEVICE_ID`. Initially diagnosed as
  "feedback dropped, grid never redraws on Theoria." **That is wrong.** The app
  mirrors LED output addressed to `lp_dev_id` onto the Theoria surface
  (`main.rs:297–312`), and kerygma broadcasts it. **Verified post-session:
  track-select pushes 309 LED cells to a headless Theoria client; reconnect
  replays 232–248 cells.** So the grid *does* redraw on select/mode change. The
  hardcoded id is fragile/ugly (and generates the F3 "device 101 dropped" spam
  as a dropped duplicate), but it is **not** the cause of the poor UX. Keep as a
  cleanup, not a keystone. The real keystone is **F8 (discoverability)**: the
  driver simply could not *find* the track-select control; once pointed at the
  top-bar numbered buttons, "it works."
- F8 (**bug/UX**): changing the sequence-edit track on Theoria is only possible
  via the transport numbered buttons (they synthesize SHIFT+pad); there is no
  SHIFT on glass and nothing labels the buttons as track-select-for-sequencing.
  Driver could not discover it. Compounded by F7 (no feedback) and by
  `/script/lp/selected` **not being mirrored** — so the client's `mirroredSelected`
  is always undefined and the button highlight is local-only (won't reflect
  selection made from another surface).
- F9: default selected track is index 0 and sequence mode edits only the
  selected track; with no visible selection state, the surface reads as
  "only one track is sequenceable."

## Verdict (driver)

**"Technically 10/10. UX total trash. It's gotta be something Roland or Elektron
would ship — right now it's Behringer."**

The pipe is proven; the instrument is not built. Every path routes (selection,
sequencing, contextual encoders, live-trigger) at ~14 ms RTT, but the surface is
unusable without a code-side guide: track-select is undiscoverable, mode is
invisible, and switching gives no feedback. That is the single most important
output of the session.

## Roadmap delta

**P10 C2–C5 ordering stands** — the pattern engine is server-side and independent
of the tablet UX; it can proceed. **But the W-track re-orders:** a dedicated
*Theoria legibility* phase must precede any further W-feature work. The paired-
session cadence should not resume until the tablet is legible enough to evaluate
features rather than fight the shell.

Minimum bar for that phase (all traceable to findings above). Ordered by the
CORRECTED diagnosis — **discoverability first, LED plumbing is a cleanup**:
1. **Track-select as a first-class, labeled control** (keystone, F8/F9) — the
   driver could not find it. Not a synthesized SHIFT+pad hidden among transport
   buttons; a clearly-labeled track selector showing which track is active and
   its name.
2. **Visible mode** — trigger vs sequence, always on screen (F2). Needs
   `/script/lp/mode` reaching the client (currently `/script/*` is not mirrored).
3. **Selected-track ↔ grid legibility** — the grid redraws correctly (verified);
   what's missing is a label saying *which* track these 16 steps belong to.
4. **Track count driven by welcome topology**, not a hardcoded 8 (F5).
5. Cleanup (not keystone): stop hardcoding `LP_DEVICE_ID`; route feedback to the
   originating surface so the app-level mirror hack (`main.rs:297`) can retire
   (F7). Fixes the F3 log spam too.
6. Nice-to-have: a semantic pattern channel so a client can render steps without
   reverse-engineering LED colors (F4); on-glass save/reload.

**QA constraint:** every item above is *visual* and must be judged against the
Elektron bar on the actual tablet — it cannot be verified headlessly. This is
paired/visual work, not blind implementation.

Quality bar for the phase: **Elektron/Roland-grade legibility, not Behringer.**
Saved as a standing design principle (memory: theoria-ux-bar).
